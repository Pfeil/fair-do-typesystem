# Uncertainty: 0.FDO/Name Localized Value Structure

## Issue

The specification states that `0.FDO/Name` provides "Human-readable and localized labels" but does not explicitly define:

1. **How localization is encoded** - Is it an object with `value` and `lang` fields? A language-tagged string like `"Name"@en`?
2. **The exact JSON structure** when multiple languages are provided

## Current Implementation

Currently using a simple string syntax reference (`0.FDO/StringSyntax`). This allows validation but doesn't enforce proper localization structure.

## Possible Approaches

### Option A: Object with value and lang
```json
"0.FDO/Name": [
  {
    "value": "FDO Type",
    "lang": "en"
  },
  {
    "value": "Typ",
    "lang": "de"
  }
]
```

### Option B: Language-tagged keys
```json
"0.FDO/Name": {
  "en": "FDO Type",
  "de": "Typ"
}
```

### Option C: Inline combination of value + lang syntaxes
Use `InlineCombination` validation mechanism with separate syntax definitions for the value string and language code.

## Resolution Needed

The specification should clarify:
- Whether there's a canonical format for localized strings
- How implementations should handle missing translations
- Whether language codes follow a specific standard (e.g., BCP 47)
